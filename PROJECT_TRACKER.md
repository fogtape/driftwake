# 漂流生存 Web 3D 项目追踪

> 工作名：`Driftwake`（暂定）  
> 项目目录：`raft-survival-web`  
> 私有仓库：`https://github.com/fogtape/driftwake`  
> 建档日期：2026-07-15  
> 当前阶段：M3 背包、合成与生存
> 当前状态：`DOING`

## 1. 项目结论

这是一个面向桌面浏览器的第一人称 3D 海上生存游戏，不按“基础 Demo”标准开发。目标是在原创品牌、原创资产和原创叙事的前提下，让前期生存循环、交互密度、环境氛围和系统咬合度接近成熟商业生存游戏。

首个完整版本以单人体验为主，必须打通以下循环：

`漂流观察 -> 抛钩打捞 -> 管理物资 -> 解决饥渴 -> 扩建木筏 -> 抵御袭击 -> 停靠岛屿 -> 水下采集 -> 研究升级 -> 前往更远海域`

完成度不能只按“按钮能点、数值会变”判断。每个核心系统必须同时具有：

- 可理解的世界反馈；
- 连贯的第一人称动画；
- 对应的环境音、动作音和 UI 音；
- 明确但不过量的 UI 提示；
- 可保存、可恢复的状态；
- 在目标设备上的稳定性能；
- 与整体画风一致的最终或近最终资产。

## 2. 产品边界

### 2.1 本期目标

- 桌面 Chrome / Edge 最新稳定版，键鼠操作；
- 全屏、第一人称、单人游玩；
- 程序化海面、天气、昼夜与漂流物带；
- 木筏扩建、承重表现、破坏和维修；
- 打捞、背包、快捷栏、合成、研究与工具耐久；
- 饥饿、口渴、生命、氧气和状态效果；
- 净水、烧烤、钓鱼、种植和基础烹饪；
- 鲨鱼巡游、预兆、攻击、驱赶、受伤与战利品；
- 小型岛屿、岸上采集、水下礁区和返回木筏的风险；
- 航帆、锚、风向和后续导航装置；
- 分版本存档、设置、暂停、失败与恢复流程；
- 完整声音分层、交互反馈和基础无障碍设置。

### 2.2 暂不作为首发门槛

- 在线联机与服务器权威同步；
- 大型剧情岛和完整长篇叙事；
- 移动端触控游玩；
- 大规模 MOD 系统；
- 原作全部中后期设备与内容数量。

这些内容不是永久删除，而是在单人核心体验达到质量门槛后进入扩展阶段。移动设备访问时仍需提供布局完整的能力提示页，不能出现黑屏或错位 UI。

### 2.3 原创与版权边界

- 不使用 `Raft` 名称、Logo、角色、剧情文本、专有地点名或营销素材；
- 不抓取、反编译、描摹或重新分发原作模型、贴图、动画、字体和音频；
- 不逐像素复制原作 HUD、菜单、图标和配色；
- 可以研究同类玩法的节奏、信息层级和系统关系，再以原创视觉与实现表达；
- 所有第三方资产必须记录来源、许可证、作者和修改情况；
- 发布前生成 `THIRD_PARTY_NOTICES`，未确认授权的资产不得进入正式包。

## 3. 体验支柱

### P1：海面必须像一个空间，而不是蓝色地板

- 远近波浪尺度分离，木筏、漂浮物和玩家有一致的升沉关系；
- 天空、雾、太阳、云影、反射、折射、泡沫和水下色散共同工作；
- 平静、起风、阴雨和风暴需要改变可见度、音景、漂流速度与操作风险；
- 地平线必须稳定，镜头摆动不能造成无意义眩晕。

### P2：手上的每个动作都要有重量

- 抛钩包含蓄力、预瞄、抛出、绳索张力、入水、命中、回收和收获反馈；
- 拾取、开箱、放置、修理、饮水、进食、钓鱼和近战均有独立动作节奏；
- 交互目标通过轮廓、材质响应或世界提示表达，不能只靠屏幕中央文字；
- 动作、音效、粒子、镜头和 UI 数值变化必须在同一时刻闭环。

### P3：资源压力推动选择，而不是拖延时间

- 玩家前几分钟能理解木板、塑料、纤维类资源各自的用途；
- 饥渴速度允许观察和试错，但会持续制造优先级冲突；
- 建造、工具、食物和防御争夺同一批资源；
- 岛屿和水下资源提供高收益，同时让玩家暴露于漂筏、缺氧和鲨鱼风险。

### P4：木筏既是载具，也是玩家作品

- 建造网格稳定、预览清晰，旋转、替换、取消与批量连续建造顺手；
- 放置物真正附着于木筏坐标系，不因波浪产生穿模或漂移；
- 鲨鱼破坏外围结构会影响设备安全，维修和加固有明确价值；
- 光照、摆件、收纳和结构升级让木筏逐步从残骸变成家。

### P5：安静与危险交替出现

- 平静漂流、收集整理和落日时刻需要留白；
- 鲨鱼袭击、缺水、暴风雨和岛屿探索形成短时压力峰值；
- 危险到来前必须有可学习的声音与视觉预兆；
- 不依靠频繁弹窗、任务播报或持续战斗维持刺激。

## 4. 前期流程目标

以下是首个纵向切片需要呈现的 30 至 60 分钟流程。具体配方和数值会经过实测重新平衡，不照抄其他游戏的数据。

1. 玩家在很小的木筏上醒来，只有简易钩具和少量应急物资。
2. 观察漂流物方向，抛钩回收木料、塑料、纤维、箱桶和稀有废料。
3. 制作绳索、杯具、简易净水器、简易烤架、建造锤和木矛。
4. 在饥渴压力下选择继续打捞、钓鱼、扩建或储备防御物资。
5. 第一次鲨鱼啃咬木筏，玩家通过声音预兆定位并用木矛驱赶。
6. 扩建外围结构，放置收集网、储物箱、作物盆和航帆。
7. 发现小岛，使用一次性锚停靠；采集树木、水果、石料和植物。
8. 潜入浅礁采集金属、沙、黏土、海草等进阶资源，同时管理氧气和鲨鱼距离。
9. 回到木筏建造研究台，投入材料解锁更耐用工具、熔炼和高级生活设备。
10. 完成第一轮金属升级，木筏具备持续生存与向更远海域航行的能力。

纵向切片通过的标志不是第 10 步能够触发，而是玩家在无开发者解释的情况下能理解选择、完成流程，并愿意继续扩建。

## 5. 质量门槛

### 5.1 画面

- 最终画风：明亮但不幼稚的风格化写实，形体概括、材质有手绘层次；
- 色彩不能被单一蓝色统治：海洋冷色与木材、锈蚀金属、植物、食物和危险提示形成层级；
- 核心物品必须使用可辨识模型与材质，几何体占位只允许存在于明确标注的技术阶段；
- 工具、双手、绳索、水花、鱼线、鲨鱼和建造预览在第一视角中不能穿帮；
- 后处理克制使用，优先保证水面可读性、交互目标辨识和夜间可玩性。

### 5.2 交互

- 常用动作从输入到首帧反馈目标低于 100ms；
- 同一按键在同一上下文中只表达一个主要动作；
- 长按行为必须显示进度并支持取消；
- 放置失败要说明空间、材料或结构原因；
- 背包、快捷栏、工作台和容器之间支持鼠标操作与常用快捷操作；
- 暂停、失焦、Pointer Lock 退出和浏览器返回场景必须可恢复。

### 5.3 声音

- 音景至少分为海浪、风、木结构、天气、生物、交互、UI 和音乐层；
- 波浪与风随天气和相对运动变化，不能使用一条短循环覆盖全场；
- 鲨鱼接近、咬筏和扑咬拥有不同的预兆与冲击层；
- 同类高频动作采用多样本、音高和音量轻微随机，避免机械重复；
- 提供主音量、音乐、环境、音效和 UI 独立控制，以及静音失焦策略。

### 5.4 性能与稳定性

- 参考目标：桌面 1920x1080、合理中端独显、默认画质稳定 60 FPS；
- 低画质目标：1280x720 稳定 30 FPS；
- 首屏只加载进入菜单所需资源，世界资产按阶段加载；
- 使用实例化、对象池、LOD、压缩纹理和合批控制漂流物与植被成本；
- 每个里程碑记录 draw calls、三角面、纹理内存、JS heap 和帧耗时；
- 存档升级失败不得损坏原存档，崩溃前后状态必须可诊断。

### 5.5 每个系统的完成定义

一个系统只有同时满足以下条件才能标为 `DONE`：

- 主路径与至少两个边界场景可运行；
- 数据、表现、音效、UI 和存档接通；
- 有自动化测试覆盖纯逻辑部分；
- 有 Playwright 流程或可重复的手工验收步骤；
- 桌面截图通过视觉检查，移动视口没有错位或内容重叠；
- 没有未登记的临时资产、版权不明资源或阻断级错误；
- 追踪文档已更新证据和遗留项。

## 6. 技术方向

技术选型会在 M0 原型验证后锁定，当前首选如下：

| 层 | 首选方案 | 目的 |
| --- | --- | --- |
| 工程 | Vite + TypeScript | 快速开发、明确类型与浏览器构建 |
| 3D | Three.js | 场景、材质、渲染与资源管线 |
| 物理 | Rapier | 玩家碰撞、射线、触发器与动态物体 |
| UI | React | 菜单、HUD、背包、合成与设置界面 |
| 状态 | 领域 Store + 事件总线 | 隔离渲染状态、模拟状态和 UI 状态 |
| 音频 | Web Audio API 封装 | 分组混音、空间音频与动态环境层 |
| 资产 | glTF/GLB + Draco/Meshopt + KTX2 | 模型与纹理压缩、流式加载 |
| 存档 | IndexedDB + 版本化 Schema | 大状态持久化与迁移 |
| 测试 | Vitest + Playwright | 逻辑测试、流程测试和截图回归 |

架构原则：

- 固定时间步推进生存、AI、天气和木筏状态，渲染帧只做插值；
- 木筏使用局部坐标系，建筑、设备和玩家接触状态围绕同一根节点计算；
- 配方、物品、战利品、天气与岛屿生成尽量数据驱动；
- 视觉表现监听领域事件，不让粒子或 UI 成为玩法真值来源；
- 高频对象使用池化，禁止在主循环中持续创建临时对象；
- 存档只保存可重建的领域状态，不序列化 Three.js 对象。

## 13. 追踪规则

- 状态只使用 `BACKLOG`、`TODO`、`READY`、`DOING`、`BLOCKED`、`DONE`；
- 一次只允许一个主要里程碑处于 `DOING`；
- 每次开发结束必须更新当前迭代、系统总表和变更日志；
- `DONE` 必须附代码位置、测试命令、截图或可复现验收步骤；
- 新需求先写入对应里程碑或 `BACKLOG`，确认优先级后再实现；
- 发现技术债时登记影响和偿还节点，不用模糊的“以后优化”；
- 任何占位资产都要带清理任务和截止里程碑。
