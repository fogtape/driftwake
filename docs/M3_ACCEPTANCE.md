## 已形成的闭环

- `drowning`、`dehydration`、`starvation`、`shark`、`injury` 五类失败原因；
- 生命归零后切换到 `failed`，固定步、输入、放置、面板和 Pointer Lock 同步停用；
- 全部工具、一个应急水、一个口粮和一个空杯受保护，其余非工具物资按确定性 35% 规则散落；
- 散落物在动态木筏最右侧外沿生成一份池化世界回收包，可继续近拾或钩取；
- v12 保存失败原因、散落清单、发生时刻和 `dropPending`，只在世界落物成功后清除待处理标记；
- 全屏失败 UI 显示原因、实际损失、保留项和恢复数值；
- 恢复清空移动、跳跃、水流冲量、镜头震动和临时交互，玩家回到筏面 `(0, 1.08)`；
- 恢复后生命/水分/饱食/氧气为 `62/44/48/100`，筏体、设备、研究和世界掉落保持原状；
- 恢复先进入稳定暂停页，再由“继续漂流”单独申请 Pointer Lock；
- 失败与恢复具有独立程序音效层，鲨鱼致命伤保留明确死因。
- 聚合库存继续作为库存真值，新增确定性堆叠投影，不引入无法持久化的格位顺序；
- 通用不可变转移函数同时负责容量预判与提交结果，返回请求量、尝试量、实际移动量以及目标已满/部分容纳原因；
- 干舱支持一个、半组、整组、逐个增减、双击整组和鼠标拖放，背包与干舱共用同一条显式数量转移链；
- 打捞钩仍被强制保留在随身工具位，目标无容量时提交禁用，部分容量时显示实际可容数量；
- 打开背包、研究台或干舱时不再同时渲染暂停模态层；模拟暂停后的 3D 呈现限制为 4 FPS，为低端设备和 DOM 交互保留主线程预算；
- 980×680、640×720 和 320×720 三档布局均无横向溢出，极窄视口使用四列物品格与纵向滚动。
- 28 个配方具有显式制作时长，制作页支持逐项数量步进并最多排入八项；材料按每个队列项原子提交，不用一次调用瞬间产出；
- 队首以 0.1 秒确定性子时钟接入现有固定步，暂停、失焦、页面隐藏、Context Lost 或 Pointer Lock 退出时与世界模拟共同冻结；
- 取消会尝试完整返还该项材料，背包容不下或升级工具发生同类冲突时拒绝取消，不会部分返还或吞物资；
- 成品背包无位或同类唯一工具占位时停在 100%，条件解除后继续结算；升级矛/斧会保留被消耗工具的耐久以供取消，并在完成时给新工具满耐久及延续快捷栏选择；
- 制作目录与队列采用 710/270 双栏工位，显示批量成本、总时长、活动进度、等待/阻塞原因与返料可用性；入队、完成、取消使用三组独立程序音效层；
- v13 保存队列稳定 ID、配方、进度、旧工具耐久和选择迁移意图，清洗非法项、重复 ID、重复唯一工具和越界进度，迁移 v1-v12 时建立空队列。

2026-07-18 结果：

- 32 个测试文件、179 项测试通过；
- TypeScript 与生产构建通过；
- 领域测试覆盖稳定堆叠投影、奇数半组、精确部分转移、容量限制、不变性和源/目标守恒；运行时测试覆盖暂停呈现间隔；
- `advanced` 最终构建浏览器场景实际把 6 个漂木拆半存入 3 个、拖放 20 个、再取回 1 个，最终背包为 4，干舱为 `20 + 10`，容量从 7/8 变为 8/8；满舱时废铁正确预判仅可容 2 个，无既有堆叠的生鱼提交被禁用；
- 制作领域的 8 项新测试覆盖 28 配方时长、批量逐项备料、队列容量与部分成功、研究/唯一工具、跨项时间结转、满包 100% 等待、取消原子返料和存档清洗；Store 测试覆盖升级工具取消耐久恢复与完成换代；
- `crafting` 最终构建浏览器场景实际排入 3 条编织绳、核对纤维 `14 -> 8`，取消两条后核对 `8 -> 10 -> 12`，恢复 Pointer Lock 后按固定步完成余下一条，最终绳索 `5 -> 6`、运行时/v13 队列同时归零；
- 1440×900 高画质制作合成图已人工检查双栏工位、批量成本和两条队列，无文本、按钮或滚动区遮挡；1024×640 双栏与 640×720 上队列/下目录布局均执行面板边界、分区衔接及横向溢出门禁；
- 干舱打开后一秒内始终只有一个 `aria-modal` 对话框、无暂停页重叠、`contextHealthy=true`；桌面面板 980×680，两侧无横向溢出，640×720 面板边界保持在视口内；
- 最终构建的浏览器场景稳定验证失败阶段 `simulationActive=false`、无 Pointer Lock；
- 最终构建的 v12 将 `3 漂木 + 2 聚合片 + 1 应急水` 结算为一份右舷回收包，`dropPending` 只从 `true` 变为 `false` 一次；
- Store 与存档测试验证恢复后失败记录清空、玩家回到筏面、恢复值正确且世界回收包仍存在；
- 浏览器脚本已编码 1440x900 UI 边界、恢复存档和暂停页断言，但当前 Termux 在恢复点击或 WebGL readback 阶段随机结束 GLES 进程/丢失 Context，未形成最终合成视觉证据。

## 目标真实 GPU 门禁

```sh
CAPTURE_ONLY=failure CAPTURE_QUALITY=high npm run capture
CAPTURE_ONLY=failure CAPTURE_QUALITY=low npm run capture
CAPTURE_ONLY=advanced CAPTURE_QUALITY=high npm run capture
CAPTURE_ONLY=advanced CAPTURE_QUALITY=low npm run capture
CAPTURE_ONLY=crafting CAPTURE_QUALITY=high npm run capture
CAPTURE_ONLY=crafting CAPTURE_QUALITY=low npm run capture
```

人工连续检查：

1. 鲨鱼、水下溺水、脱水和饥饿四条真实触发路径都只进入一次失败页；
2. 海面、木筏和失败 UI 的构图层级清楚，没有白屏、穿模、文本溢出或按钮遮挡；
3. 水下到筏面的恢复没有错误曝光、海面翻转、相机埋板或单帧强烈跳变；
4. 右舷回收包在高低浪位均可见、可近拾、可钩取，且不会落在筏板下方；
5. 失败低频、原因噪声、恢复提示和随后暂停音景的响度衔接自然；
6. 失败时刷新、恢复前刷新、恢复后刷新均不会复制或删除散落物；
7. 920x1600 桌面模式与 1280x720 低画质下内容仍完整，不与浏览器安全区重叠。
8. 拆分、双击和拖放在 30/60 FPS 下均只提交一次，容量不足反馈与真实移动量一致，转移音效方向明确。
9. 批量入队、取消返料和顺序完成在 30/60 FPS 下均只提交一次；入队、完成、取消三组音效可辨且不盖过海面与危险提示。

## 当前环境限制

Termux Chromium 可稳定完成失败结算、DOM、v12 一次性落物、`advanced` 的真实拆分/拖放/取回，以及 `crafting` 的 v13 入队、取消、恢复和完成闭环。暂停呈现限频后，软件 GLES 下的容器 DOM 不再被连续 3D 帧饿死；1440×900 制作页已取得 CDP 合成证据，但其他含 WebGL 的高分辨率整页截图或 `ReadPixels` 仍可能卡住合成器或结束 GLES 进程，不能替代目标真实 GPU 的帧率、失败画面与最终混音门禁。
